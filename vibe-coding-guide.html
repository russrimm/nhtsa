  • Node.js 20+ installed — nodejs.org (verify with node -v in your terminal)

Step 1 — Plan & Build
1
Research, plan, then trigger the build
Give the AI a URL and a quality bar — it asks the right questions, then builds everything
Prompt A — kick off research
Your exact prompt
I want to utilize all of the useful APIs from NHTSA website at https://www.nhtsa.gov/nhtsa-datasets-and-apis and build an automotive analyzer app in typescript and react on node.js. Research the apis and determine the next steps. I want it to look modern and I want it to be useful for anyone who uses it.
✓ What happened
The agent fetched the NHTSA API documentation page, identified 6 usable public APIs (vPIC decode, NCAP Safety Ratings, Recalls, Complaints, Investigations, Car Seat Locator), and asked 4 clarifying questions before writing any code.

The 4 clarifying questions — and the answers given in this session:

  • Framework: Vite + React SPA (no server-side rendering)
  • UI style: shadcn-style components with Tailwind CSS and Radix UI primitives
  • Features: All available features (VIN decode, ratings, recalls, complaints, investigations, car seat locator)
  • Deployment: Local development only (no backend or cloud hosting required)
Prompt B — trigger the build
Your exact prompt
Start implementation.
✓ What happened
~40 files were created in one pass: package.json, vite.config.ts, tsconfig.json, Tailwind config, all API client modules, all React Query hooks, all page components, all UI primitives (Button, Card, Input, Badge, Tabs, Dialog, Select, Skeleton), routing, dark/light theme system, recent-search localStorage persistence, MapLibre map, and a Recharts chart. The agent ran npm install automatically. Two TypeScript errors that surfaced during the build were fixed automatically — no additional prompts needed.
💡 Key lesson
The plan is the real work. Giving the AI a URL to research (rather than describing APIs from memory) means it works from accurate documentation. The clarifying Q&A produces a written plan that acts as a contract — then two words trigger the entire build.
Step 2 — Fix Problems As They Surface
2
Paste errors, describe wrong output, report regressions
Every problem is just another prompt — paste it exactly, never summarize

Problems fall into three types. All three happened in this build and all three were fixed with a single prompt each.

Type 1 — Paste a build error or warning verbatim
Prompt (CSS @import build error — pasted exactly from terminal)
transforming (4) src\App.tsx[vite:css] @import must precede all other statements (besides @charset or empty @layer) 55 | 56 | /* MapLibre styles */ 57 | @import 'maplibre-gl/dist/maplibre-gl.css';
✓ What happened
The MapLibre CSS @import was placed after @tailwind directives in src/index.css. CSS requires all @import rules first. The agent moved it to the top. One-file fix.
Prompt (bundle size warning — pasted exactly from build output)
vite v5.4.21 building for production... ✓ 2551 modules transformed. dist/assets/index-DxvSvOUt.js 1,655.20 kB │ gzip: 470.37 kB (!) Some chunks are larger than 500 kB after minification. Consider: - Using dynamic import() to code-split the application - Use build.rollupOptions.output.manualChunks to improve chunking
✓ What happened
All page components were converted to React.lazy() with Suspense, and vite.config.ts was updated with manualChunks to split MapLibre, Recharts, React, TanStack Query, and Radix UI into separate vendor bundles. Initial load dropped from 470 KB → ~80 KB gzipped.
Prompt (runtime error — pasted exactly from browser error overlay)
Unexpected Application Error! Button is not defined ReferenceError: Button is not defined at VehicleHubPage (http://localhost:5173/src/pages/VehicleHubPage.tsx?t=1779196911206:85:11)
✓ What happened
The Button component was used in VehicleHubPage.tsx but the import was missing. The agent added import { Button } from '@/components/ui/button';. One-line fix.
Type 2 — Describe wrong UI output (attach a screenshot)
Prompt (with screenshot attached)
the screen says lookup by year/make/model but when selecting the make it should say things like Honda, Hyundai, Ford, Toyota, etc but it shows this instead
✓ What happened
NHTSA's GetAllMakes endpoint returns every registered manufacturer (~12,000 entries including custom shops like "#1 Alpine Customs"). The agent switched the dropdowns to the recall products API (/products/vehicle/makes?modelYear={year}&issueType=r), which returns only mainstream consumer brands scoped to the selected year.
Type 3 — Report a regression (when a fix breaks something new)
Prompt
now it's showing "No image available" on each one I search
✓ What happened
NHTSA returns make/model in ALL CAPS (HYUNDAI, PALISADE). Wikipedia article titles use Title Case. The previous fix introduced a casing mismatch that caused 404s on every lookup. The agent added a toTitleCase() step before constructing Wikipedia search titles. One-line fix.
💡 Key lesson
Never summarize an error — paste it exactly. Build errors, browser stack traces, and build warnings are already precise machine-readable descriptions of the problem. Paraphrasing removes the information the agent needs. Screenshots work the same way for visual bugs.
Step 3 — Add Features By Describing the Experience
3
Describe what the user should experience — not how to build it
The agent chooses the architecture; you choose the outcome
Example A — asking if something is possible
Prompt
is there any way to consistently display an image of the vehicle selected?
✓ What happened
The agent evaluated free image sources (no API key required), chose the Wikipedia MediaWiki API as the most reliable option, created src/lib/api/vehicleImage.ts with a filename-scoring algorithm to avoid irrelevant images, built a VehicleImage React component with loading skeleton and car-icon fallback, and added image display to both the VIN result page and the vehicle hub page.
Example B — requesting a full new feature
Prompt
i would like to add a compare feature so you can select multiple cars and show all of the values for each car to compare them
✓ What happened
Eight files were created or modified: localStorage persistence store, React Context (add / remove / clear / isInCompare), a sticky bottom tray showing selected vehicles with a "Compare →" button, a full comparison page with NCAP star ratings and green/red highlights for best/worst per category, a new lazy route, a nav link, and an "Add to Compare" toggle button on the vehicle hub page.
💡 Key lesson
Describe the user experience — not the implementation. "Is there any way to…" lets the agent evaluate feasibility. "I would like to add a feature so you can…" communicates intent. State management, context API, routing, and component architecture are the agent's responsibility — not yours.

The mindset shift

Traditional coding: you write instructions for a computer.
Vibe coding: you write intent for an AI. The AI writes the instructions for the computer.

Your job is to be precise about what you want and honest about what went wrong. The agent's job is to figure out how.
